In `ai_chat_scroll_view.dart`, in the `build()` method:

1. Wrap the existing `NotificationListener<ScrollMetricsNotification>` (from Plan 01) and `CustomScrollView` with an additional `NotificationListener<ScrollUpdateNotification>`:

```dart
NotificationListener<ScrollUpdateNotification>(
  onNotification: (notification) {
    if (notification.dragDetails != null && _anchorActive) {
      _anchorActive = false;
      // No setState — _anchorActive is control state only, not render state.
      // _onScrollChanged will now no-op because _anchorActive is false.
    }
    return false; // NEVER absorb — consumer app may listen to scroll notifications
  },
  child: NotificationListener<ScrollMetricsNotification>(
    // ... existing from Plan 01
    child: CustomScrollView(
      // ... existing
    ),
  ),
)
```

The nesting order matters: `ScrollUpdateNotification` fires when the user drags. We check `dragDetails != null` which is `true` if and only if the scroll was caused by a physical drag gesture (not a programmatic `jumpTo`). This is the same pattern used by Flutter's `RefreshIndicator`.

After implementation, run all tests. API-04 tests should turn GREEN. All ANCH tests from Plan 01 should still pass.

IMPORTANT: Do NOT use `setState` to flip `_anchorActive` — it is internal control state that does not affect the rendered widget tree directly. The `_fillerHeight.value` changes (via ValueNotifier) are the only thing that produces visual output, and those are already guarded by the `_anchorActive` check.
